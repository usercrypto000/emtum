# Verifier Gas Baseline
*May 2026*

The Merkle inclusion circuit now has an EVM verifier generated by `bb.js` against the validated circuit artifact. Public inputs remain only `policy_root` and `action_hash`; the private leaf preimage and Merkle path stay inside the witness.

Foundry uses `optimizer = true` with `optimizer_runs = 1` because higher optimizer runs leave runtime bytecode above the EIP-170 maximum contract size of 24,576 bytes (Ethereum, EIP-170, 2016). Under that deployable profile, verifier runtime bytecode is 24,253 bytes and one valid inclusion proof verifies at 2,164,576 gas (Local Foundry run, May 3 2026). The proof fixture is 6,848 bytes with two public inputs, so calldata is roughly 7,044 bytes before transaction envelope costs (Local fixture generation, May 3 2026).

A 200-run optimizer profile gives cheaper verification at 2,149,489 gas, but runtime bytecode is 25,183 bytes, which is not mainnet deployable without verifier splitting, compiler work, or circuit changes (Local Foundry run, May 3 2026). The simulation should use the deployable `optimizer_runs = 1` profile until a smaller verifier path is available. This confirms the production constraint: single-proof verification is expensive enough that aggregation remains a production milestone, not a cosmetic optimization.
